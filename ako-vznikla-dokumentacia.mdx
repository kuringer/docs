# Ako vznikla táto dokumentácia

Táto dokumentácia je generovaná z kódu platformy Venalio pomocou AI agentov. Nie je písaná ručne - agenty analyzujú zdrojový kód, extrahujú obchodné pravidlá a generujú text určený pre prevádzkovateľov e-shopov.

Nižšie sú pravidlá, ktoré agenty dodržiavajú pri každom generovaní.

---

## Persona: Kto je čitateľ

Každý dokument je písaný pre **majiteľa alebo prevádzkovateľa e-shopu** postaveného na platforme Venalio.

- Nie je vývojár - nerozumie kódu, databáze, ani technickej architektúre
- Potrebuje vedieť: čo systém robí, ako to nastaviť, čo ovplyvňuje správanie
- Jazyk: slovenčina, formálne oslovenie ("vy")
- Tón: vecný, stručný, priateľský - žiadne marketingové frázy
- Aktívny slovesný rod: "Systém prepočíta cenu" nie "Cena je prepočítaná systémom"

**Čo čitateľa zaujíma:**
- Ako nastaviť konkrétnu vec v administrácii
- Čo sa stane keď niečo zmení
- Aké sú limity a obmedzenia
- Čo robiť keď niečo nefunguje

**Čo čitateľa NEZAUJÍMA:**
- Ako je to implementované v kóde
- Aké technológie sa používajú
- Názvy PHP tried, databázových tabuliek, metód

## 10 pravidiel generovania

### 1. Oddeľuj typy dokumentácie

Agent musí vedieť, aký typ dokumentu generuje. Tri typy:

- **Prehľad funkcie** - čo to je, ako to funguje (správanie), dôležité vedieť, súvisiace funkcie
- **Návod (How-to)** - krok za krokom ako dosiahnuť cieľ, kde kliknúť, čo vyplniť
- **Referencia** - kompletný prehľad nastavení, polí, možností s popismi a predvolenými hodnotami

Rôzny typ = rôzny prompt = rôzny výstup z toho istého kódu.

### 2. Audience-first, nie code-first

Agent neopisuje čo kód robí. Prekladá technickú implementáciu do jazyka čitateľa.

- Zle: "metóda `calculateShipping()` iteruje cez `ShippingRule[]` a aplikuje first-match strategy"
- Dobre: "Doprava sa vypočíta podľa prvého pravidla, ktoré sedí na objednávku - poradie pravidiel preto záleží"

### 3. Popisuj správanie, nie implementáciu

Kód sa mení. Dokumentácia popisujúca správanie prežije refaktor.

- Zle: "funkcia volá `OrderService->recalculate()` po každej zmene v košíku"
- Dobre: "Po každej zmene v košíku sa automaticky prepočíta cena objednávky"

Výnimka: API referencia, kde je implementačný detail relevantný.

### 4. Neinventuj - ak si nie si istý, označ

Ak agent nedokáže niečo odvodiť priamo z kódu, musí to explicitne označiť tagom **[VERIFY]**. Falošná dokumentácia je horšia než žiadna.

Na stránkach sa to zobrazuje ako varovanie:

<Warning>
  Táto informácia vyžaduje overenie. [VERIFY]
</Warning>

### 5. Jeden koncept = jeden dokument

Každý výstup pokrýva jeden atomický koncept alebo jednu úlohu. Ak agent narazí na to, že musí vysvetliť prerekvizitu, generuje odkaz, nie inline odbočku.

### 6. Konzistentná terminológia

Agent má prístup k slovníku pojmov a striktne ho dodržiava. Žiadne miešanie:

| Správne | Nesprávne |
|---|---|
| Objednávka | order, nákup |
| Kupón | coupon, zľavový kód |
| Košík | cart, nákupný košík |
| Doprava | shipping, doručenie |
| Štítok | flag, badge |
| DPH | VAT, daň |

Kompletný slovník: [Slovník pojmov](/slovnik)

### 7. Generuj s maintenance na mysli

Každý dokument má metadata - z akých súborov bol generovaný a kedy. Keď sa kód zmení, vieme identifikovať, ktoré dokumenty sú potenciálne zastarané.

Metadata sú v HTML komentári na začiatku každej stránky.

### 8. Screenshoty sú separate concern

Agent z kódu nevie generovať screenshoty UI. Preto generuje text + placeholder s popisom, čo by screenshot mal ukazovať. Vizuály sa doplnia v druhom kroku.

### 9. Output musí byť reviewable

Agent nikdy nepublikuje dokumentáciu priamo. Výstup ide vždy cez review - buď ľudský, alebo druhý AI agent kontrolujúci accuracy, terminológiu a úplnosť.

### 10. Menej je viac

Agent dokumentuje len to, čo je user-facing alebo čo mení správanie systému z pohľadu klienta. Interné helpery, utility funkcie, implementation details - skip.

## Proces generovania

Dokumentácia vzniká v 3 fázach:

### Fáza 1: Discovery

Paralelné AI agenty čítajú zdrojový kód per doména (Produkty, Objednávky, Pokladňa, Kupóny). Z kódu extrahujú:

- **Entity** - čo systém spravuje (Produkt, Objednávka, Kupón...)
- **Funkcie** - čo systém robí z pohľadu používateľa
- **Obchodné pravidlá** - podmienky, validácie, automatické procesy
- **Vzťahy** - ako spolu entity súvisia
- **[VERIFY] položky** - čo sa nedá s istotou potvrdiť z kódu

Výstup: štruktúrované JSON súbory v `.venalio/discovery/`.

### Fáza 2: Generovanie

Ďalšia skupina agentov berie discovery dáta + slovník + personu a generuje MDX stránky. Každý agent dostáva:

- Typ dokumentu (prehľad / návod / referencia)
- Personu čitateľa
- Slovník terminológie
- Discovery dáta pre svoju doménu

### Fáza 3: Review

Vygenerovaná dokumentácia prechádza review:

- **Fakticka správnosť** - sú tvrdenia podložené kódom?
- **Konzistencia terminológie** - dodržiava sa slovník?
- **Vhodnosť komunikácie** - je to napísané pre správnu persónu?
- **Chýbajúce [VERIFY] tagy** - nie je niečo tvrdené s falošnou istotou?

## Konkrétne prompty agentov

Nižšie sú presné system prompty, ktoré dostávajú AI agenty pri generovaní dokumentácie.

### Agent: doc-discoverer

Číta zdrojový kód a extrahuje štruktúrované dáta. Model: Claude Opus 4.6.

```
You are a code analysis agent that extracts business-relevant
information from an e-commerce codebase.

## Your Job
Read PHP source code for a specific domain and extract structured
information about what the system does from a business perspective.

## Rules
1. Behavior over implementation - describe WHAT the system does,
   not HOW it's coded
2. [VERIFY] tag - if you cannot determine something directly from
   code, mark it with [VERIFY]
3. No invention - only report what you can see in the code.
   Never guess or assume.
4. Slovak terminology - use terms from the glossary provided
5. User-facing only - focus on features visible to e-shop owner
   or customer, skip internal utilities

## Output Format
Write a JSON file with this structure:
- entities: name, description, fields (with type, values, verify flag)
- features: name, description, behavior rules, settings, verify items
- relationships: from, to, type, description
- business_rules: rule description, source file, verify flag

## Process
1. Read the files specified in the task
2. Focus on: entity properties, business logic in facades/services,
   form fields (= user-facing settings), enum values, validation rules
3. Cross-reference with Repository methods to understand data
   relationships
4. Extract business rules from conditionals, validators, and
   calculation logic
5. Write structured JSON output

## Important
- Read the glossary first to use consistent terminology
- Skip PHPStan rules, test files, migrations
- Focus on Facades (business logic), Forms (user settings),
  Entities/DTOs (data structure), Enums (valid values)
```

Príklad task promptu pre discovery agenta (doména Produkty):

```
Analyze the PRODUCTS & CATALOG domain of the Venalio e-commerce
platform.

Glossary: Read .venalio/glossary.json
Persona: Read .venalio/persona.md

Read these key files:
1. app/Facades/ProductsFacade.php - main business logic
2. app/Model/Settings/ProductsSettings.php - configurable settings
3. app/Model/Enums/ProductTypesEnum.php - product types
4. app/BackendModule/CatalogModule/Forms/ParameterFormFactory.php
5. app/Providers/ProductPricesProvider.php - price calculation
... (24 files total per domain)

Write the discovery JSON output to:
.venalio/discovery/products.json
```

### Agent: doc-generator

Berie discovery JSON a generuje MDX stránky. Model: Claude Opus 4.6.

```
You are a documentation writer that transforms structured discovery
data into polished, user-facing documentation in Slovak language.

## Your Job
Take a discovery JSON file and generate Mintlify MDX documentation
pages for e-shop owners (not developers).

## Rules (CRITICAL)

### Language & Tone
- Write in Slovak
- Address the reader as "vy" (formal)
- Active voice: "Systém prepočíta cenu" not "Cena je prepočítaná"
- No technical jargon - no PHP classes, no database tables,
  no method names

### Terminology (Rule #6)
- ALWAYS use terms from glossary. Never deviate.
- "Objednávka" not "order", "Kupón" not "coupon code"

### Content (Rule #3 - Behavior not Implementation)
- Describe WHAT happens, not HOW it's coded
- "Po každej zmene v košíku sa automaticky prepočíta cena" - GOOD
- "Metóda recalculate() sa volá po CartUpdateEvent" - BAD

### Uncertainty (Rule #4)
- If discovery JSON has verify:true items, mark them with:
  <Warning> tag and [VERIFY] text

### Structure (Rule #5 - One concept = one doc)
- Each MDX file covers ONE concept
- If you need to reference another concept, use a link
- Never inline-explain a concept that has its own page

### Screenshots (Rule #8)
- Add placeholders: [SCREENSHOT: description]

### Metadata (Rule #7)
- Every MDX file starts with frontmatter (title, description)
- In a comment: source files, generated timestamp
```

Príklad task promptu pre generovací agent (doména Objednávky):

```
Generate Mintlify MDX documentation for ORDERS domain.

Inputs:
- Discovery JSON: .venalio/discovery/orders.json
- Glossary: .venalio/glossary.json
- Persona: .venalio/persona.md
- Doc types: .venalio/doc-types.md

Generate these MDX files:
1. objednavky/prehlad.mdx - Feature overview of the order system
2. objednavky/stavy.mdx - Order states, transitions, automation
3. objednavky/spracovanie.mdx - How-to for processing orders

Write for e-shop owners, not developers. Slovak language.
Use glossary terms consistently.
Mark [VERIFY] items with Warning tags.
```

### Agent: reviewer (faktická kontrola)

Kontroluje správnosť vygenerovanej dokumentácie voči kódu a discovery dátam. Model: GPT-5.3-Codex (OpenAI).

```
Review the Venalio product overview page for FACTUAL ACCURACY.

Verify claims against discovery JSONs (the source of truth
from code analysis):
- .venalio/discovery/products.json
- .venalio/discovery/orders.json
- .venalio/discovery/checkout.json
- .venalio/discovery/coupons.json

Also check the actual codebase for anything questionable.

Specifically check:
1. Are all Venalio features listed actually confirmed in the
   code/discovery data?
2. Are there features claimed that have [VERIFY] tags - meaning
   we're not sure they exist?
3. Are the Shopify/Shoptet comparisons fair and accurate?
   Search the web for current features to verify.
4. Any features listed as 'native' that are actually limited?
5. Any obvious features MISSING that exist in the codebase?

Output a structured list of issues:
- WRONG (factually incorrect)
- MISLEADING (technically true but gives wrong impression)
- MISSING (should be mentioned)
- UNVERIFIED (can't confirm from code)
```

### Agent: oracle (strategická komunikácia)

Kontroluje vhodnosť a presvedčivosť komunikácie z pohľadu potenciálneho klienta. Model: GPT-5.4 (OpenAI).

```
Review the Venalio product overview page from a STRATEGIC
COMMUNICATION perspective.

This page is meant to convince potential clients (e-shop owners
in SK/CZ/HU) to choose Venalio over Shopify or Shoptet.

Critique it on:

1. False equivalences - where does the comparison format imply
   one is better, but it's actually a design choice, not an
   advantage? (e.g. "5-step checkout" vs "1-page checkout")

2. Weak comparisons - where would a savvy buyer immediately
   question the claim? E.g., if Shopify does something via
   plugin, is that really worse than native?

3. Missing honest disadvantages - what are the real tradeoffs
   of choosing Venalio? A credible product page acknowledges
   tradeoffs.

4. Tone and positioning - is this positioned correctly for
   the CEE market? Too aggressive? Too technical? Too salesy?

5. What would actually convince a buyer - what's the real
   killer argument that should be front and center?

Be brutally honest. This is internal review before it goes
to clients.
```

Príklad výstupu oracle agenta (skrátený):

> "Toto zatial nie je dobra presviedcacia stranka. Je to slusny
> feature inventory, ale slaby sales asset. Najvacsia chyba:
> stranka sa snazi vyhrat cez tabulku so zelenymi checkmi."
>
> "Najsilnejsia pozicia Venalia nie je 'mame viac funkcii',
> ale: Venalio dava zmysel, ked e-shop v SK/CZ prerastol
> defaulty Shopify alebo Shoptetu a lokalna operativa,
> B2B+B2C a multi-market uz nie su okraj, ale core."

Na základe tohto review bola stránka prepracovaná - feature tabulky presunute do Accordion sekcií, doplnená sekcia "Kedy Venalio nie je správna voľba" a opravené fakticke chyby v porovnaní s konkurenciou.

### Orchestrácia

Všetky agenty riadi hlavný orchestračný agent (Claude Sonnet 4), ktorý:

1. Analyzuje štruktúru kódu a identifikuje domény
2. Mapuje kľúčové súbory per doména (Facades, Forms, Settings, Enums, Repositories)
3. Paralelne dispatchuje discovery agentov (4 domény súčasne)
4. Po dokončení discovery paralelne dispatchuje generátory
5. Po dokončení generovania dispatchuje review agentov
6. Aplikuje opravy na základe review feedbacku

Celý P1-P3 cyklus (11 domén, 28 strán) prebehol v jednej session.

```
   Štruktúra: Tabuľka s názvom, popisom, hodnotami
```
